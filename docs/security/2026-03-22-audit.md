# 安全審計報告 — Local AI Freelance Agency OS

**審計日期：** 2026-03-22
**審計版本：** Phase 3.4
**審計範圍：** `src/`（全部）、`package.json`（依賴漏洞）、`.dispatch/`（IPC 檔案處理）
**審計方式：** 唯讀靜態程式碼分析（Read-only static analysis）
**整體風險評級：** 🟡 中等風險（需修復高風險項目後方可上線）

## 執行摘要

Agency OS 是一個設計良好、測試覆蓋率高（118/118）的 TypeScript 應用程式，架構清晰。程式碼本身品質優良，無明顯注入或路徑遍歷漏洞。然而，有一個**高風險憑證曝露問題**以及數個**缺少身份驗證**的 API 端點，需在正式上線前優先修復。

## 一、OWASP Top 10 風險審查

### [高風險] A01:2021 — Broken Access Control（存取控制失效）

**漏洞描述：**
`/api/dispatch` 及 `/api/intake` 兩個 HTTP 端點完全未實作身份驗證或授權機制。任何能夠連線到伺服器的人均可：
- 建立任意任務（`POST /api/dispatch`）
- 取消任意任務（`DELETE /api/dispatch/tasks/:id`）
- 提交任意答案至 IPC 問答流程（`POST /api/dispatch/tasks/:id/answer`）
- 觸發 Anthropic API 呼叫（消耗 API Token 費用）

**影響範圍：**
`src/modules/dispatch/api.ts`、`src/modules/intake/api.ts`

**修復建議：**
- 短期：限制伺服器僅監聽 `127.0.0.1`（localhost），避免對外暴露
- 中期：實作 API Key 驗證（`X-API-Key` Header），驗證比對環境變數 `AGENCY_API_KEY`
- 長期：評估是否需要 JWT 或 OAuth 2.0 授權流程

---

### [高風險] A05:2021 — Security Misconfiguration（安全設定錯誤）

**漏洞 1 — 硬編碼 API 金鑰（CWE-798）：**

`.mcp.json` 檔案以明文儲存 Context7 API 金鑰：

```
"CONTEXT7_API_KEY": "ctx7sk-f11fc597-888e-46b8-87cb-fc1f05b327faY"
```

此檔案已被納入 git 版本控管（非 `.gitignore`），導致金鑰曝露於：
- 本機磁碟（任何可存取工作目錄的程序皆可讀取）
- Git 歷史紀錄（即使未來刪除，歷史紀錄仍保有該金鑰）
- 若推送至 GitHub，則為公開曝露

**影響範圍：** `.mcp.json`

**修復建議（立即執行）：**
1. 前往 Context7 控制台撤銷並重新產生 API 金鑰
2. 將新金鑰設定為環境變數：`export CONTEXT7_API_KEY=<new-key>`
3. 修改 `.mcp.json` 改用環境變數方式引用
4. 將 `.mcp.json` 加入 `.gitignore`（若包含任何憑證）

**漏洞 2 — 缺少 CORS 標頭：**
HTTP 回應不含 `Access-Control-Allow-Origin` 標頭，任何來源的跨域請求均被接受。

**影響範圍：** 兩個 API 模組的所有回應

**修復建議：** 明確設定 CORS 為 `localhost` 或受信任來源

---

### [中風險] A04:2021 — Insecure Design（不安全設計）

**漏洞描述：**
缺少請求速率限制（Rate Limiting）及請求大小限制（Content-Length 上限）。

**具體風險：**
- 攻擊者可透過大量 `POST /api/intake` 請求，消耗 Anthropic API Token（經濟損失）
- 超大型請求（如 100MB JSON）可能導致記憶體耗盡（DoS）
- 無速率限制使暴力掃描 task_id 成為可能

**影響範圍：** `src/modules/dispatch/api.ts`、`src/modules/intake/api.ts`

**修復建議：**
- 加入 Content-Length 上限檢查（建議 1MB）
- 實作基於 IP 的速率限制（建議每分鐘 10 次）
- 設定請求逾時（建議 30 秒）

---

### [中風險] A07:2021 — Authentication Failures（身份驗證失敗）

**同 A01，另補充：**
IPC 問答流程（`.dispatch/tasks/:slug/ipc/*.question` / `.answer`）依賴檔案系統隔離作為唯一「授權」機制。任何具有本機檔案系統讀寫權限的程序均可偽造答案、注入惡意 `.question` 檔案或刪除 `.done` 標記，破壞任務流程。

**影響範圍：** `src/modules/dispatch/ipc-manager.ts`、`.dispatch/tasks/` 目錄

**修復建議：**
- 對 IPC 檔案內容加入 HMAC 簽名驗證（防止偽造）
- 明確設定 `.dispatch/tasks/` 目錄的檔案權限為 `0700`（僅擁有者可讀寫執行）

---

### [中風險] A09:2021 — Security Logging and Monitoring Failures（日誌與監控失效）

**漏洞描述：**
目前 `src/utils/logger.ts` 僅提供極簡的 `console.log/error` 封裝，不記錄：
- HTTP 請求的來源 IP、時間戳、方法、路徑
- 任務建立、取消、完成等狀態轉換
- Anthropic API 呼叫紀錄（模型、Token 用量）
- 驗證失敗嘗試

**影響範圍：** `src/utils/logger.ts`、所有 API 端點

**修復建議：**
- 改用結構化日誌（Winston 或 Pino）
- 記錄每筆 HTTP 請求（IP、方法、路徑、狀態碼、回應時間）
- 記錄所有安全相關事件（驗證失敗、異常請求）

---

### [低風險] A03:2021 — Injection（注入）

**審查結果：✅ 通過**

- **SQL Injection：** 不適用（系統為純檔案系統架構，無資料庫）
- **Command Injection：** 無 `exec`/`spawn` 呼叫接受使用者輸入
- **Path Traversal：** `src/utils/slug.ts` 的 slug 生成函式有效過濾 `/`、`..` 等危險字元，路徑遍歷風險已控制

**slug 安全審查：**
```typescript
slug = slug.replace(/[^\x00-\x7F]/g, "");    // 移除非 ASCII
slug = slug.replace(/[^a-z0-9 -]/g, "-");   // 特殊字元替換為 -
slug = slug.replace(/^-+|-+$/g, "");         // 修剪首尾 -
```
此實作有效防止路徑遍歷。

---

### [低風險] A08:2021 — Software and Data Integrity Failures

**審查結果：✅ 通過（含說明）**

`src/utils/file-writer.ts` 使用原子寫入（Atomic Write）：
```typescript
const tmpPath = `.tmp_${Date.now()}_${base}`;
await fs.writeFile(tmpPath, content);
await fs.rename(tmpPath, filePath);  // 原子性 rename
```

此設計防止了部分寫入導致的資料損毀。惟檔案權限預設（通常 `0644`）略為寬鬆。

**修復建議（低優先）：**
```typescript
await fs.writeFile(tmpPath, content, { mode: 0o600 }); // 僅擁有者可讀寫
```

## 四、API 端點輸入驗證、認證與授權審查

| 端點 | 輸入驗證 | 認證 | 授權 |
|------|---------|------|------|
| `POST /api/dispatch` | ✅ alias 白名單驗證 | ❌ 無 | ❌ 無 |
| `GET /api/dispatch/tasks` | ✅ 分頁參數驗證 | ❌ 無 | ❌ 無 |
| `GET /api/dispatch/tasks/:id` | ✅ UUID 格式隱性驗證 | ❌ 無 | ❌ 無 |
| `POST /api/dispatch/tasks/:id/answer` | ✅ 非空驗證 | ❌ 無 | ❌ 無 |
| `DELETE /api/dispatch/tasks/:id` | ✅ 任務存在性檢查 | ❌ 無 | ❌ 無 |

**正面發現：**
- Alias 白名單限制（僅允許 13 個預定義別名），有效防止任意任務建立
- JSON 解析錯誤處理完善
- HTTP 狀態碼使用正確（201/200/400/404/500）

**[中風險] 錯誤訊息資訊洩漏：**

```typescript
// src/modules/dispatch/api.ts (約第 149 行)
send(res, 500, { status: "error", message: (err as Error).message });
```

原始錯誤訊息直接回傳給用戶，可能洩漏：
- 系統內部路徑
- 堆疊追蹤
- 實作細節

**修復建議：**
```typescript
// 正式環境應回傳通用錯誤訊息，詳細錯誤僅記錄至日誌
logger.error('Unhandled error', { err });
send(res, 500, { status: "error", message: "Internal server error" });
```

## 五、IPC 路徑遍歷與注入風險審查

### IPC 管理器分析 (`src/modules/dispatch/ipc-manager.ts`)

**IPC 目錄路徑構成：**
```typescript
const ipcDir = (taskSlug: string) =>
  path.join(CONFIG.paths.tasks, taskSlug, "ipc");
```

**路徑遍歷風險評估：**

`taskSlug` 的來源路徑：
1. 用戶輸入 `project_name` → `buildIntakeSlug()` → 嚴格清理 → **安全**
2. `POST /api/dispatch` 的 `alias` 參數 → 白名單驗證 → **安全**
3. 任務 UUID → 系統生成 → **安全**

**[通過] 路徑遍歷風險：已有效控制**

`slug.ts` 的清理函式確保 slug 不含 `/`、`..` 或任何路徑分隔字元，路徑遍歷攻擊向量已封閉。

**[低風險] IPC 檔案內容未驗證完整性：**

`.answer` 檔案的內容被直接讀取並使用，未進行：
- 簽名驗證（任何程序均可偽造 `.answer` 檔案）
- 內容格式驗證（僅檢查非空）
- 來源驗證

在本機使用情境下風險較低，但若系統在多租戶或網路環境中部署則需特別注意。

**[低風險] Handlebars 模板注入風險：**

`src/modules/dispatch/template-engine.ts` 使用 Handlebars 進行模板渲染：
```typescript
const template = Handlebars.compile(templateContent);
const rendered = template(context);
```

**風險：** 若 `templateContent` 來源可被用戶控制，Handlebars 歷史上有 Prototype Pollution 漏洞。但本專案中模板檔案由開發者維護（`.dispatch/templates/`），用戶無法控制模板內容，風險極低。
